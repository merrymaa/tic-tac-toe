# Архитектура приложения

## Обзор

Это веб-приложение для игры в крестики-нолики, построенное с использованием **Flask** и следующее принципам **чистой архитектуры** с разделением на слои.

## Структура проекта

```
src/
├── main.py                 # Точка входа в приложение
├── constants.py            # Константы приложения
├── web/                    # Веб-слой (Presentation Layer)
│   ├── app.py              # Создание Flask-приложения
│   ├── route/              # HTTP-маршруты (контроллеры)
│   ├── model/              # DTO для веб-слоя
│   ├── mapper/             # Мапперы между слоями
│   └── module/             # Вспомогательные модули (аутентификация, контроллеры)
├── domain/                 # Доменный слой (Business Logic)
│   ├── model/              # Доменные модели
│   └── service/            # Интерфейсы доменных сервисов
├── datasource/             # Слой работы с данными (Infrastructure Layer)
│   ├── database/           # Работа с базой данных
│   ├── model/              # Модели данных (ORM)
│   ├── repository/         # Репозитории для доступа к данным
│   ├── service/            # Реализации сервисов
│   └── mapper/             # Мапперы между моделями
├── di/                     # Dependency Injection
│   └── container.py        # Контейнер зависимостей
└── test/                   # Тесты
```

## Слои архитектуры

### 1. Web Layer (`web/`)
**Ответственность:** Обработка HTTP-запросов и формирование ответов.

- **`route/`** — определяет конечные точки API:
  - `auth_routes.py` — маршруты аутентификации (регистрация, вход)
  - `game_routes.py` — маршруты игровых операций (создание игры, ходы, получение состояния)

- **`model/`** — DTO (Data Transfer Objects) для передачи данных между клиентом и сервером
- **`module/`** — вспомогательные компоненты:
  - `ControllerWeb` — координация между веб-слоем и сервисами
  - `UserAuthenticator` — проверка токенов пользователей

### 2. Domain Layer (`domain/`)
**Ответственность:** Бизнес-логика приложения, независимая от фреймворков и БД.

- **`model/`** — чистые доменные модели (например, `Game`, `User`, `Field`)
- **`service/`** — интерфейсы бизнес-сервисов (контракты)

### 3. Datasource Layer (`datasource/`)
**Ответственность:** Работа с источниками данных (база данных).

- **`database/`** — инициализация и подключение к БД
- **`model/`** — ORM-модели для хранения данных
- **`repository/`** — репозитории для CRUD-операций
- **`service/`** — конкретные реализации сервисов (например, `UserServiceImpl`, `RepositoryBackedService`)
- **`mapper/`** — преобразование между доменными моделями и моделями данных

### 4. Dependency Injection (`di/`)
**Ответственность:** Управление зависимостями между компонентами.

- **`container.py`** — единый контейнер, создающий и хранящий экземпляры сервисов:
  - `game_service` — сервис для управления играми
  - `user_service` — сервис для управления пользователями

## Поток запроса

1. **HTTP-запрос** поступает на маршрут в `web/route/`
2. **Маршрут** вызывает соответствующий метод через `ControllerWeb`
3. **Контроллер** использует сервис из `di.container`
4. **Сервис** (`datasource/service/`) выполняет бизнес-логику
5. **Репозиторий** (`datasource/repository/`) работает с базой данных
6. **Ответ** проходит обратно через слои и возвращается клиенту

## Примеры эндпоинтов

| Метод | Эндпоинт | Описание |
|-------|----------|----------|
| POST | `/new_game` | Создать новую игру |
| GET | `/get_games` | Получить список активных игр |
| GET | `/get_current_game` | Получить состояние конкретной игры |
| POST | `/make_move` | Сделать ход в игре |
| POST | `/join` | Присоединиться к игре |
| GET | `/get_user` | Получить информацию о пользователе |

## Запуск приложения

```bash
cd src
python main.py
```

Сервер будет доступен по адресу: `http://localhost:5000`

Примеры запросов находятся в: `src/test/curl.txt`
